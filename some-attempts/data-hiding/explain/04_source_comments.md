# 04 — 源代码注释/字符串载体（Comments & String Litter）

对应样本：`attack/04_source_comment_carrier_SIMULATED.cpp`

## 这类“攻击”在做什么
攻击者把“隐藏信息”伪装成：
- 很长的注释
- 看似无害但从未使用的字符串常量
- 高熵常量/魔法数
- 冗余代码块（永远不执行的分支）

从代码评审角度，它们可能被误认为调试残留或无关文本。

> 本样本仅用于展示“可疑形态”，不包含可还原的秘密或编码方法。

## 典型风险场景
- **内鬼/供应链**：恶意提交把信息塞进源码，随后通过公开仓库或包发布传播。
- **审阅疲劳**：评审时注意力集中在业务逻辑变更，忽略“文本噪声”。

## 可观测信号
- 新增大量无意义注释或未引用常量。
- 大量高熵字符串（类似随机 key 的形态）。
- 可疑文件在多个 PR 中逐步累积“噪声”。

## 防护与缓解
- **静态分析**：检测未使用常量/未使用变量/死代码。
- **评审规范**：限制注释长度与用途；对高熵字符串要求来源说明。
- **发布前扫描**：对源码包做敏感信息与异常文本扫描。

## 练习建议
- 用 lint 规则找出未使用的 `kLooksRandomButIsNot` 风格常量。
- 给告警添加解释：为什么它可疑、建议如何处置。
