# 02 — 空白通道（Whitespace Channel in Text/Markdown）

对应样本：`attack/02_whitespace_channel_SIMULATED.md`

## 这类“攻击”在做什么
攻击者利用空白字符（空格、制表符、换行、缩进差异等）在视觉上“看起来一样”的文本中携带信息。常见载体是 Markdown、代码片段、配置文件、邮件正文。

> 本仓库样本是**模拟**：用 `[TAB]`、`[SPACE]` 这类可见 token 表示“空白差异”，不使用真实可编码的尾随空白序列。

## 典型风险场景
- **绕过文本审查**：DLP/关键字扫描通常会忽略或规范化空白，导致“隐藏信道”不被关注。
- **供应链/审阅绕过**：代码评审时不容易注意到尾随空格或 tab 差异。

## 可观测信号
- 行尾空白、tab 与 space 混用异常。
- 文件 diff 中出现“看似无改动但实际有改动”的情况。
- 单个文件的空白模式与该仓库/团队格式化规范不一致。

## 防护与缓解
- **强制格式化**：提交前自动 trim trailing spaces、统一缩进（editorconfig/formatter）。
- **提交策略**：CI 中拒绝包含行尾空白或混用 tab/space 的变更。
- **审计工具**：在 diff 视图中可视化空白（显示 whitespace changes）。

## 练习建议
- 用 lint/预提交钩子检测 `[TAB]` / 多余空白标记。
- 测试“规范化后再比较”的检测策略是否能消除这类载体。
